## 1. 왜 Vercel을 떠나는가

### Phase 1이 Vercel이었던 이유

Phase 1의 목표는 "일단 돌아가는 것을 보여주는 것"이었다. 단일 HTML에 Vercel 서버리스 함수 몇 개를 붙이면 그날 안에 배포가 된다. 서버도, 컨테이너도, 클러스터도 필요 없다. 프로토타입 단계에서 이건 명백히 옳은 선택이었다.

문제는 Phase 3의 목표가 **클라우드 네이티브 / MSA / Kubernetes**라는 점이다. 그리고 그 목표는 Vercel 위에서는 도달할 수 없다.

### Vercel 서버리스의 한계

| 필요한 것 | Vercel 서버리스에서 왜 안 되나 |
|-----------|------------------------------|
| 상시 구동 서버 | 함수는 요청이 올 때 깨어나고 끝나면 죽는다. 상태를 들고 있을 수 없다 |
| DB 커넥션 풀 | 함수가 매번 새로 뜨니 연결을 재사용하지 못한다. 커넥션이 계속 새로 열린다 |
| 서비스 분리(MSA) | 함수 단위 분리는 되지만, 서비스별 독립 배포·스케일링·롤링 업데이트는 안 된다 |
| 장기 실행 작업 | 실행 시간 제한이 있다. 배치·수집 작업을 돌릴 수 없다 |

> 🏢 **비유**: Vercel 서버리스는 **필요할 때만 부르는 대리기사**다. 부르면 오고, 목적지에 내려주면 사라진다. 편하지만, 이 차를 24시간 세워두고 계속 짐을 실어두거나 정비할 수는 없다. Phase 3이 요구하는 건 대리기사가 아니라 **차고와 정비소를 갖춘 우리 차**다.

특히 DB 연결이 결정적이었다. 화성형 팔란티어는 민원·피드·센서 데이터를 PostgreSQL에 쌓고 그 위에서 분석하는 시스템이다. 커넥션 풀을 유지하지 못하는 실행 모델 위에서는 이 구조 자체가 성립하지 않는다.

### 왜 하필 네이버 클라우드인가

AWS도 GCP도 있는데 네이버 클라우드로 간 이유는 두 가지다.

**첫째, 국내 데이터 주권.** 화성형 팔란티어가 다루는 건 시민의 민원 데이터다. 공공기관 시스템에서 데이터가 어느 나라 관할의 물리 서버에 저장되는지는 기술 선택이 아니라 **규제 요건**에 가깝다. 국내 리전에 국내 사업자가 운영하는 인프라라는 조건이 협상 대상이 아니었다.

**둘째, 네이버 API 생태계와의 일치.** 이 시스템은 이미 네이버 검색 API로 뉴스·블로그·카페를 수집하고, 네이버 지도 API로 시각화한다. 데이터 수집과 시각화가 이미 네이버 생태계 안에 있는데 컴퓨트만 다른 클라우드에 두는 건 경계를 하나 더 만드는 일이다.

## 2. 네이버 클라우드 인프라 구축기

### VPC 설계 — 서브넷 5개

VPC(`hspalantir-vpc`, `192.168.0.0/16`)를 만들고, 그 안을 용도별로 5개 서브넷으로 쪼갰다. 전부 KR-2 존이다.

| Subnet | 대역 | 유형 | 역할 |
|--------|------|------|------|
| `hspalantir-public` | 192.168.1.0/24 | Public / 일반 | 외부에 노출되는 일반 리소스 |
| `hspalantir-private` | 192.168.2.0/24 | Private / 일반 | 내부 전용. **DB 이전 예정** |
| `hspalantir-nks` | 192.168.3.0/24 | Private / NKS 노드 | K8s 워커 노드가 사는 곳 |
| `hspalantir-lb-private` | 192.168.4.0/24 | Private / LoadBalancer | 로드밸런서 전용 |
| `hspalantir-nat-gw` | 192.168.5.0/24 | Public / NAT Gateway | NAT Gateway 전용 |

> 🏢 **비유**: VPC가 **분양받은 단지 부지 전체**고, 서브넷은 그 안을 용도별로 나눈 **구역**이다. 주거동 구역, 상가 구역, 주차장 구역, 정문 구역을 처음부터 갈라놓는 것과 같다. 나중에 섞으려면 도로를 다시 깔아야 한다.

핵심은 **용도별 분리**다. 노드와 로드밸런서를 같은 서브넷에 두면 당장은 돌아가지만, IP가 모자라거나 방화벽 규칙이 충돌할 때 손댈 수 없는 상태가 된다.

### Cloud DB for PostgreSQL

`Cloud DB for PostgreSQL (G3)`로 `hspalantir-db`를 만들었다.

```
Host:     pg-47dq25.vpc-pub-cdb-kr.ntruss.com
Port:     5432
Database: hspalantir
User:     hsadmin
```

지금은 **Public 도메인 방식**이다. 로컬 개발 PC에서 바로 붙어서 스키마를 만들고 데모 데이터를 넣어야 했기 때문이다. 스키마는 5개다.

| 테이블 | 용도 |
|--------|------|
| `complaints` | 민원 데이터 |
| `feeds` | 뉴스·블로그·카페 수집 |
| `analysis_logs` | AI 분석 로그 |
| `dispatch_records` | 맑은물사업소 긴급출동 기록 |
| `sensor_readings` | 수압 센서 데이터 |

**단, 이건 임시 상태다.** Public 도메인은 인터넷에서 이 DB에 접근할 경로가 열려 있다는 뜻이다. `hspalantir-private` 서브넷을 미리 파둔 이유가 여기 있다 — 운영 전환 시 DB를 Private으로 옮기고 클러스터 내부에서만 접근하게 막아야 한다. 지금 편하자고 만든 구조를 그대로 운영에 가져가면 안 된다.

### NAT Gateway — 삽질 기록

Private 서브넷에 있는 노드는 **외부 인터넷으로 나갈 수 없다.** 그런데 나가야 한다. 컨테이너 이미지를 받아야 하고, 네이버·Claude·Gemini API를 호출해야 한다.

들어오는 건 막고 나가는 건 되게 하는 물건이 NAT Gateway다.

> 🏢 **비유**: NAT Gateway는 **단지 정문의 택배 대리수령 창구**다. 주민(Private 노드)이 밖으로 주문은 넣을 수 있지만, 외부인이 그 창구를 통해 주민 집 주소를 알아내 찾아올 수는 없다. 나가는 길만 있고 들어오는 길은 없다.

**여기서 삽질했다.** NAT Gateway를 만들려는데 서브넷 선택 목록에 아무것도 안 떴다. 이미 만들어 둔 Public 서브넷이 있는데도 그랬다.

원인은 서브넷의 **용도(Usage Type)** 였다. 네이버 클라우드는 서브넷을 만들 때 "일반"과 "NAT Gateway 전용"을 구분한다. 나는 `hspalantir-nat-gw`를 만들면서 용도를 "일반"으로 지정했고, NAT Gateway는 "NAT Gateway 전용" 서브넷만 받는다. **서브넷 용도는 생성 후 변경할 수 없어서**, 서브넷을 지우고 용도를 바꿔 다시 만들었다.

배운 것: 네이버 클라우드에서 서브넷은 `Public/Private` × `용도` 두 축으로 결정되고, **둘 다 나중에 못 바꾼다.** 만들기 전에 이 서브넷에 뭘 넣을지 확정해야 한다.

최종 흐름은 이렇다.

```
Private 노드 (192.168.3.0/24)
    ↓  기본 경로 0.0.0.0/0
NAT Gateway (공인 IP 223.130.131.157)
    ↓
외부 인터넷 (NCR, Naver API, Claude API, Gemini API)
```

### Route Table

NAT Gateway를 만들어놓기만 하면 아무 일도 안 일어난다. **"외부로 나갈 땐 저리로 가라"고 길을 알려줘야** 한다.

`hspalantir-vpc-default-private-table`에 규칙을 넣었다.

| 목적지 | 다음 홉 |
|--------|--------|
| `0.0.0.0/0` | NAT Gateway |

> 🏢 **비유**: NAT Gateway를 세운 게 "정문에 택배 창구를 지은 것"이라면, Route Table은 **단지 안 곳곳에 붙이는 "외부 우편물은 정문 창구로" 안내판**이다. 창구만 있고 안내판이 없으면 주민은 그런 게 있는 줄도 모른다.

### NKS 클러스터

마지막으로 클러스터를 만들었다. 자세한 설정 항목은 지난 두 편에서 다뤘으므로 결과만 정리한다.

| 항목 | 값 |
|------|-----|
| 클러스터 | `hspalantir-cluster` |
| Kubernetes | 1.35.3 |
| 존 | 단일 존 (KR-2) — 검증 단계라서. 운영 시 다중 존 재검토 |
| CNI | cilium |
| 노드풀 | `hspalantir-nodepool`, `s2-g3a` (2vCPU/8GB) × 2노드, 100GB |

프로비저닝은 10여 분 만에 에러 없이 끝났다. **네트워크를 먼저 다 깔아놓고 클러스터를 마지막에 만든 순서** 덕분이다. 클러스터 생성이 실패하는 원인은 대부분 클러스터가 아니라 그 앞의 네트워크에 있다.

## 3. Docker 컨테이너화 및 NCR Push

### Vercel 함수를 Express로 감싼 이유

Vercel에서 `api/` 폴더의 각 `.js` 파일은 그 자체로 하나의 엔드포인트다. 라우팅을 Vercel이 알아서 해준다. 컨테이너로 옮기면 그 "알아서"가 사라진다. **라우팅을 직접 해줄 서버가 필요하다.**

그래서 `server.js`를 만들어 기존 함수 4개를 Express 라우트로 연결했다.

| 기존 (Vercel) | 컨테이너 (Express) | 역할 |
|--------------|-------------------|------|
| `api/search.js` | `/api/search` | 네이버 검색 API 프록시 |
| `api/extract.js` | `/api/extract` | Claude 장소 추출 |
| `api/agent.js` | `/api/agent` | AI 분석 에이전트 |
| `api/water-agent.js` | `/api/water-agent` | Gemini + PostgreSQL 수도민원 분석 |

**함수 내부 코드는 거의 그대로 뒀다.** 목적은 리팩터링이 아니라 이전이다. 실행 환경을 바꾸면서 코드까지 바꾸면, 문제가 생겼을 때 원인이 환경인지 코드인지 알 수 없게 된다. 한 번에 하나씩만 바꾼다.

> 🏢 **비유**: Vercel 함수 4개가 **각자 알아서 손님을 받던 노점 4개**였다면, Express 서버는 그 4개를 한 건물 안에 넣고 **정문에 안내 데스크를 세운 것**이다. 파는 물건은 그대로고, 손님을 어느 창구로 보낼지 안내하는 사람만 새로 생겼다.

### Dockerfile

`node:20-alpine` 기반으로 작성했다. alpine을 고른 이유는 이미지 크기다. 노드가 이미지를 당길 때마다 NAT Gateway를 통해 나가므로, 이미지가 작을수록 배포가 빠르고 트래픽 비용도 준다.

### Docker Desktop 설치 삽질

로컬에서 빌드하려고 Docker Desktop을 깔았는데 두 번 막혔다.

**1) WSL2 미설치 오류**

Docker Desktop이 뜨자마자 WSL2가 없다고 거부했다. Windows용 Docker는 사실상 리눅스 커널 위에서 도는데, 그 커널을 WSL2가 제공한다.

```powershell
wsl --install
```

**2) BIOS 가상화 미활성화**

WSL2를 깔았더니 이번엔 가상화가 꺼져 있다고 했다. 이건 OS가 아니라 **메인보드 펌웨어 설정**이라 Windows 안에서는 못 켠다. 재부팅 → BIOS 진입 → 가상화(Intel VT-x / AMD-V) 활성화 → 저장 후 재시작으로 해결했다.

> 🏢 **비유**: 컨테이너를 "가구"라고 하면, Docker Desktop은 **가구를 들일 방**이고, WSL2는 **그 방을 떠받치는 바닥**이며, BIOS 가상화는 **건물 자체의 기초 공사 허가**다. 허가가 안 났으면 바닥도 방도 못 놓는다. 순서를 건너뛸 수 없다.

배운 것: 컨테이너는 "그냥 앱"이 아니라 **하드웨어 가상화 위에 서 있는 물건**이다. 평소엔 몰라도 되지만, 안 될 땐 이 층위를 알아야 원인을 찾는다.

### NCR (Container Registry) 생성

빌드한 이미지를 어딘가에 올려야 클러스터가 받아갈 수 있다. 그게 컨테이너 레지스트리다.

**여기서도 순서에 걸렸다.** NCR을 만들려는데 Object Storage 버킷을 먼저 요구했다. NCR이 이미지 레이어를 실제로 저장하는 곳이 Object Storage이기 때문이다. `hspalantir-registry` 버킷을 먼저 만들고 나서야 레지스트리를 만들 수 있었다.

```
Registry URL: hspalantir.kr.ncr.ntruss.com
Object Storage 버킷: hspalantir-registry
```

> 🏢 **비유**: NCR은 **물류 창고의 접수 카운터**고, Object Storage는 **실제 물건이 쌓이는 창고 부지**다. 부지 없이 카운터만 세울 수는 없다.

이 지점에서 앞의 NAT Gateway 삽질과 같은 패턴을 다시 만났다. **네이버 클라우드는 리소스마다 전제 조건이 있고, 그 전제를 안 채우면 목록에 아예 안 뜬다.** 에러 메시지가 친절하게 "이게 없습니다"라고 알려주기보다, 선택지가 비어 있는 형태로 나타난다. 목록이 비어 있으면 내 설정이 틀린 게 아니라 **앞 단계가 빠진 것**이라고 의심하는 게 빠르다.

### build + push

```
docker build → hspalantir.kr.ncr.ntruss.com/hspalantir-api:v1 → docker push ✅
```

클러스터가 당겨갈 이미지가 준비됐다.

### 노트북에서 PC 작업물을 복구한 이야기

작업을 PC에서 하다가 노트북으로 옮겼는데, **일부 파일을 git push 안 한 상태**였다. PC는 그 자리에 없었다.

그런데 그 파일들은 이미 **컨테이너 이미지 안에 들어가 있었다.** NCR에 push해둔 `hspalantir-api:v1`은 빌드 시점의 파일을 그대로 담고 있으니까.

```bash
docker pull hspalantir.kr.ncr.ntruss.com/hspalantir-api:v1
docker create --name tmp hspalantir.kr.ncr.ntruss.com/hspalantir-api:v1
docker cp tmp:/app/. ./복구경로
docker rm tmp
```

`docker create`는 컨테이너를 만들기만 하고 실행하지 않는다. 실행하지 않아도 파일시스템은 존재하므로 `docker cp`로 꺼낼 수 있다.

> 🏢 **비유**: 이사 갈 때 짐을 컨테이너에 실어 보냈는데, 원본 집 열쇠를 잃어버린 상황이다. 다행히 **컨테이너에 실린 짐 목록이 곧 그 시점의 집 상태**라, 컨테이너를 열어 짐을 도로 꺼냈다.

**다만 이건 자랑이 아니라 반성이다.** 이미지가 백업 역할을 해준 건 운이 좋았던 것이지 설계가 아니다. 이미지는 빌드 시점 스냅샷이라 그 이후 작업은 없고, `.dockerignore`에 걸린 파일도 없다. 교훈은 "docker cp로 복구할 수 있다"가 아니라 **"기기를 옮기기 전에 push한다"** 이다.

## 4. 앞으로의 개발 계획

### 이번 주 — K8s 배포 완성

클러스터는 섰지만 아직 아무것도 안 돌고 있다. 여기까지 가야 Vercel을 끌 수 있다.

| 단계 | 내용 |
|------|------|
| 1 | kubectl 설치 + NKS kubeconfig 연결 → `kubectl get nodes`로 노드 2대 Ready 확인 |
| 2 | 매니페스트 4개 작성 및 apply — `secret` / `deployment` / `service` / `ingress` |
| 3 | Ingress Controller (nginx) 설치 |
| 4 | NCP Load Balancer 연결 (`hspalantir-lb-private` 서브넷 사용) |
| 5 | HTTPS — cert-manager + Let's Encrypt |
| 6 | DNS 전환 → **Vercel 종료** |

`secret.yaml`에는 DB 비밀번호와 API 키 4종이 들어간다. **git에 커밋하지 않는다.**

### 8월 — 기능 고도화

- **GitHub Actions CI/CD**: push → build → NCR push → rolling update 자동화. 지금은 이 과정이 전부 수동이라, 배포가 무서워서 자주 안 하게 된다. 자주 배포하려면 배포가 지루해져야 한다
- **water-demo 완성**: Gemini + PostgreSQL 수도민원 원인분석 데모
- **민원 CSV 업로드 → 지오코딩 → 지도 마커**: 실제 민원 데이터를 좌표로 바꿔 지도에 얹는 파이프라인

### 10월 — MSA 전환

지금은 Express 단일 서버 안에 4개 기능이 함께 산다. 이걸 서비스별로 쪼갠다.

| 지금 | MSA 전환 후 |
|------|-----------|
| `/api/search` | `search-svc` (독립 Deployment) |
| `/api/extract` | `extract-svc` |
| `/api/agent` | `agent-svc` |
| `/api/water-agent` | `water-svc` |

여기에 API Gateway(Kong 또는 Ingress 세분화), 중앙 로깅(NCP Cloud Insight 또는 Grafana), Redis 캐시를 붙인다.

> 🏢 **비유**: 지금은 **한 건물에 부서 4개가 세 들어 사는 상태**다. 한 부서가 불나면 건물 전체가 탄다. MSA는 부서마다 건물을 따로 주는 것이다. 대신 건물 사이를 잇는 도로(API Gateway)와 전체를 보는 관제실(중앙 로깅)이 필요해진다.

**단일 서버로 시작한 건 실수가 아니다.** 쪼갤 경계를 모르는 상태에서 먼저 쪼개면, 잘못된 경계를 코드가 아니라 네트워크로 굳히게 된다. 한 덩어리로 돌려보며 경계를 확인한 뒤에 쪼갠다.

### 12월 — 정식 운영

- React + Vite 프론트엔드 (단일 HTML 졸업)
- PostgreSQL + **PostGIS** 공간 온톨로지
- 새올행정 API 연동
- 국가기초구역 히트맵
- 화성시 실제 운영 환경 이관

## 5. 전국 우수 공무원 결과물 통합 계획

### 배경

여기서부터가 이 프로젝트의 진짜 이야기다.

**화성형 팔란티어는 화성시 단독 시스템이 아니다.** 전국의 공무원들이 각자의 현장에서 만든 혁신 결과물을 **온톨로지로 연결하는 플랫폼**이다.

지금 전국에는 자기 업무를 자동화하려고 직접 도구를 만든 공무원들이 있다. 법령을 다루는 사람은 법령 도구를, 통계를 다루는 사람은 통계 도구를 만들었다. 문제는 이것들이 **각자 따로 존재한다**는 점이다. 각각은 훌륭한데, 서로 말을 걸지 못한다.

민원 하나를 제대로 분석하려면 이게 다 필요하다 — 관련 법령은 무엇인지, 통계적으로 이례적인지, 다른 지자체는 어떻게 했는지, 근거 행정문서에는 뭐라 적혀 있는지. **도구는 이미 다 있다. 연결이 없을 뿐이다.**

### 연결할 결과물들

**광진구 류승인 주무관**

| 도구 | 화성형 팔란티어에서의 역할 |
|------|--------------------------|
| 법령 MCP (`korean-law-mcp`) | 민원 분석 시 관련 법령 자동 연결 |
| LexDiff | 일상 언어 법률 질문 → 법령 원문·판례 기반 답변 |
| KOSIS MCP (`korean-stats-mcp`) | 국가통계 실시간 AI 조회 |
| 한글문서 파싱 MCP (`kordoc`) | HWP 행정문서 자동 분석 |

마지막 것이 특히 중요하다. 행정문서는 대부분 HWP다. 이걸 파싱하지 못하면 **행정 데이터의 상당 부분이 AI에게 안 보인다.**

**행안부 서호성 사무관** — 대한민국 제도 총정리(`korea100`). 정책 분석 에이전트의 컨텍스트로 활용한다.

**광양시 정솔현 주무관** — 공공정책 총정리(`klocal`). 타 지자체 유사 사례를 자동 검색·비교한다. "이 문제를 이미 푼 지자체가 있는가"는 공공 현장에서 가장 자주 나오는 질문이다.

**공공데이터포털** — 데이터목록조회 API로 화성시 관련 공공데이터를 전체 자동 탐색한다.

**화성시 전명구 주무관** *(파일 수령 후 상세 반영 예정)*

- 계약안내 플랫폼 — 예산 집행 × 민원 교차 분석
- 담배소매인 지정 플랫폼 — 생활편의시설 공간 데이터 연계

### 통합 아키텍처 구상

```
법령 MCP ────────────────┐
KOSIS MCP ───────────────┤
한글문서 파싱 MCP ─────────┤
공공데이터포털 API ─────────┤──→ 화성형 팔란티어 AI 에이전트
공공정책 총정리 ────────────┤      (Claude + Gemini)
대한민국 제도 총정리 ──────────┤           ↓
계약안내 플랫폼 ────────────┤   국가기초구역 공간 온톨로지
담배소매인 플랫폼 ───────────┘        (PostGIS)
                                       ↓
                             네이버 지도 시각화 대시보드
```

이 그림의 핵심은 **가운데가 아니라 아래**다. 여러 도구를 AI 에이전트에 물리는 것 자체는 어렵지 않다. 어려운 건 그 결과를 **하나의 공간 온톨로지 위에 올리는 것**이다.

법령은 조문 단위고, 통계는 행정구역 단위고, 민원은 좌표 단위다. 이 셋을 같은 화면에서 이야기하려면 공통 기준이 필요하다. 화성형 팔란티어는 그 기준을 **좌표 + 국가기초구역(우편번호 5자리)** 으로 잡았다. 행정동·읍면동 단위 집계를 쓰지 않는 이유다 — 행정 경계는 사람이 그은 선이라 바뀌지만, 좌표는 안 바뀐다.

> 🏢 **비유**: 도구들이 각각 **다른 언어를 쓰는 전문가**라면, 온톨로지는 **그들이 공유하는 지도**다. 변호사·통계학자·민원 담당자가 같은 지도 위에 각자 아는 것을 표시하기 시작하면, 그때부터 서로의 말을 이해하지 않고도 대화가 된다.

### 다음 단계

1. 류승인 주무관 MCP 서버를 **Claude API `mcp_servers` 파라미터로 연결** — Claude API는 원격 MCP 서버를 직접 호출할 수 있으므로, 별도 중계 서버 없이 붙는다
2. `water-agent.js` 패턴을 재사용해 **MCP별 에이전트 엔드포인트 추가** — water-agent는 이미 "LLM + 외부 데이터소스 교차확인 → 결론" 구조라, 데이터소스만 갈아끼우면 된다
3. 전명구 주무관 플랫폼 파일 수령 후 상세 통합 계획 수립

2번이 3장에서 "코드를 그대로 뒀다"고 한 결정과 이어진다. 검증된 패턴이 하나 있으면, 그 다음은 복제다.
