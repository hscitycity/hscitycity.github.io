## 발견 경위

NKS 클러스터에 파드를 배포하고 DB 연결을 테스트하던 중 SSL 에러가 났다.

처음엔 단순한 SSL 설정 문제로 봤다. `ssl: false`를 넣거나 `rejectUnauthorized: false`를 주면 넘어갈 것 같았다. 실제로 그렇게 하면 에러는 사라졌을 것이다.

**그런데 그게 함정이었다.** 에러를 없애는 것과 문제를 푸는 것은 다르다. SSL 에러는 SSL 문제가 아니라, **네트워크 경로 자체가 잘못됐다는 신호**였다.

---

## 문제 상황

파드와 DB는 같은 VPC(`192.168.0.0/16`) 안에 있었다. 파드는 `192.168.3.x`, DB는 `192.168.1.6`. 옆 서브넷이다.

그런데 DB 접속 도메인이 **Public 도메인**으로 설정되어 있었다. 그래서 트래픽이 이렇게 흐르고 있었다.

```
파드(192.168.3.x)
  → NAT Gateway
    → 인터넷
      → 공인 IP(101.79.10.213)
        → DB(192.168.1.6)
```

**같은 VPC 안에 있는 DB에 접속하면서 인터넷을 한 바퀴 돌아 나갔다가 다시 들어오고 있었다.**

![변경 전후 DB 접속 네트워크 경로 비교](/img/dev_img/db_network_path_comparison.png)

> 🏢 **비유**: 같은 아파트 **옆집에 편지를 보내면서 국제 우편으로 부친 격**이다. 편지는 도착한다. 그래서 문제를 눈치채기 어렵다. 다만 그 편지가 지구 반 바퀴를 돌면서 모르는 사람들 손을 거쳤을 뿐이다.

---

## 확인 방법

추측하지 않고 파드 안에서 직접 DNS를 조회했다. **도메인이 어떤 IP로 풀리는지가 곧 경로다.**

```bash
kubectl exec deploy/hspalantir-api -- node -e \
"require('dns').lookup('pg-47dq25.vpc-pub-cdb-kr.ntruss.com',(e,a)=>console.log('DB IP:',a))"
# 결과: DB IP: 101.79.10.213  ← 공인 IP (위험)
```

```bash
kubectl exec deploy/hspalantir-api -- node -e \
"require('dns').lookup('pg-47dq25.vpc-cdb-kr.ntruss.com',(e,a)=>console.log('DB IP:',a))"
# 결과: DB IP: 192.168.1.6  ← 사설 IP (안전)
```

같은 DB인데 **도메인에 따라 다른 IP가 나온다.** 하나는 공인 IP, 하나는 사설 IP다. 이 한 줄이 모든 걸 설명했다.

이 방식이 좋은 이유는 **파드 안에서** 확인했다는 점이다. 내 노트북에서 `nslookup`을 돌리면 내 위치 기준 결과가 나온다. 실제로 DB에 접속하는 주체는 파드이므로, **파드가 보는 것을 봐야 한다.**

---

## 왜 위험한가

만약 여기서 `ssl: false`로 에러를 덮었다면, **DB 비밀번호와 화성시 민원 데이터가 인터넷 구간을 평문으로 흘렀을 것이다.**

화성시 시민의 민원 데이터가 들어 있는 DB다. 가볍게 넘길 문제가 아니었다.

무서운 건 **그래도 잘 돌아갔을 것**이라는 점이다. 화면은 정상이고, 데이터도 나오고, 아무도 에러를 안 본다. 문제가 있다는 유일한 신호가 SSL 에러였는데, 그걸 설정 한 줄로 껐다면 신호마저 사라진다.

---

## 해결 방법

`secret.yaml`에서 `DB_HOST` **한 줄만** 바꿨다.

```
변경 전: pg-47dq25.vpc-pub-cdb-kr.ntruss.com   (공인 IP 101.79.10.213 경유)
변경 후: pg-47dq25.vpc-cdb-kr.ntruss.com       (사설 IP 192.168.1.6 직통)
```

**도메인에서 `pub` 세 글자를 빼는 것**만으로 트래픽 경로가 인터넷 경유에서 VPC 내부 직통으로 바뀐다.

| | Public 도메인 | Private 도메인 |
|---|--------------|---------------|
| 도메인 | `vpc-**pub**-cdb-kr.ntruss.com` | `vpc-cdb-kr.ntruss.com` |
| 풀리는 IP | 101.79.10.213 (공인) | 192.168.1.6 (사설) |
| 경로 | NAT → 인터넷 → 공인IP → DB | VPC 내부 직통 |
| 용도 | 외부(로컬 PC 등)에서 접속 | 같은 VPC 안에서 접속 |

Public 도메인이 처음부터 있었던 이유는 있다. **로컬 개발 PC에서 스키마를 만들고 데모 데이터를 넣어야 했기 때문**이다. 그때는 그게 맞았다. 문제는 그 설정을 **클러스터 안으로 그대로 들고 들어온 것**이다.

> 🏢 **비유**: 이사 오기 전 옆 동네에 살 때 쓰던 "우리 집 가는 길" 메모를 그대로 들고 온 셈이다. 이제 같은 건물에 사는데도 메모대로 지하철을 타고 있었다. 도메인 한 줄 바꾸는 건 **그냥 옆집 문을 두드리는 것**과 같다.

---

## 그래도 SSL은 켜두는 게 낫다

경로를 고쳤으니 이제 트래픽이 VPC 밖으로 안 나간다. 그러면 암호화는 없어도 되는가?

**아니다. 껐던 SSL이 있다면 다시 켜두는 쪽을 권한다.**

이유는 두 가지다.

**첫째, VPC는 물리적으로 격리된 망이 아니다.** 논리적으로 격리된 가상 네트워크다. 같은 물리 인프라 위에서 소프트웨어가 경계를 만들어 준다. 실무상 충분히 안전하지만, "전선이 물리적으로 따로 깔려 있다"는 뜻은 아니다.

**둘째, 이게 zero-trust의 핵심이다.** 오늘 팔란티어 Rubix 문서를 읽으면서 정리한 원칙이 정확히 이거였다.

> "every interaction between workloads must be authenticated, authorized, and logged in accordance with immutable configurations."
>
> "Encryption is rigorously enforced across every element in the environment."

**"내부망이니까 믿는다"는 개념이 없다.** 내부에서도 모든 상호작용을 인증하고 암호화한다. 내부망을 믿는 순간, 그 안에 한 번 들어온 공격자에게는 모든 게 열린다.

경로 수정과 암호화는 **둘 중 하나를 고르는 문제가 아니다.**

| | 역할 |
|---|------|
| **Private 도메인으로 경로 고정** | 데이터가 애초에 인터넷으로 안 나가게 함 |
| **SSL 유지** | 그 안에서도 내용을 못 읽게 함 |

앞의 것은 **노출 범위를 줄이는 것**이고, 뒤의 것은 **줄어든 범위 안에서도 방어하는 것**이다. 이걸 defense in depth라 부른다. 공공 데이터를 다루는 시스템이라면 둘 다 하는 게 맞다.

## 핵심 교훈

**1) NCP Cloud DB는 Public / Private 두 도메인을 제공한다**

같은 DB인데 접속 경로가 다르다. 콘솔에서 복사한 도메인이 어느 쪽인지 확인해야 한다. 기본으로 눈에 띄는 게 Public 도메인이라 무심코 쓰기 쉽다.

**2) 같은 VPC 안에서는 반드시 Private 도메인(`vpc-cdb`)을 쓴다**

성능(경로가 짧다), 비용(NAT 트래픽이 안 나간다), 보안(인터넷 구간이 없다) 셋 다 이득이다. 손해 보는 게 없다.

**3) SSL 에러가 네트워크 구성 문제를 드러냈다**

에러 메시지는 자기가 아는 것만 말한다. SSL 계층에서 난 에러라고 원인이 SSL에 있는 건 아니다.

**4) 에러를 설정으로 덮지 말고 근본 원인을 먼저 찾는다**

이게 제일 중요하다. `ssl: false`는 **에러를 없애는 게 아니라 경고등을 떼는 것**이었다. 경고등을 떼면 차는 계속 굴러가고, 고장은 그대로 남는다.

지난 6월 말 「표에 display flex를 걸면 안 되는 이유」를 쓸 때도 같은 교훈을 적었다. 그때는 "캐시 문제겠지"라고 추측하며 key를 두 번 버전업했지만 효과가 없었고, 결국 직접 렌더링해서 값을 찍어보고서야 원인을 찾았다. **추측으로 덮지 말고 직접 찍어볼 것.** 이번엔 그 교훈이 먹혔다 — 파드 안에서 `dns.lookup`을 직접 돌린 게 답을 줬다.

## 정리

| 항목 | 내용 |
|------|------|
| 증상 | 파드에서 DB 접속 시 SSL 에러 |
| 첫 추측 | SSL 설정 문제 (`ssl: false`로 덮을 뻔) |
| 실제 원인 | Public 도메인 사용 → 같은 VPC인데 인터넷 경유 |
| 확인 | 파드 안에서 `dns.lookup` — 공인 IP(101.79.10.213) 확인 |
| 해결 | `DB_HOST`를 Private 도메인(`vpc-cdb`)으로 변경 — `pub` 세 글자 삭제 |
| 추가 조치 | SSL은 유지 (defense in depth) |
| 교훈 | 에러를 덮지 말고 경로를 찍어볼 것 |

한 줄 고쳤는데 인프라 구성 전체를 다시 보게 된 버그였다. 그리고 이 버그를 알려준 건 결국 **내가 끄려고 했던 그 에러**였다.
