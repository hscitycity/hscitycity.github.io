## 왜 지금 이 화면을 뜯어보는가

지난 devlog에서 컨테이너·Docker·MSA·Kubernetes 개념을 정리했고, 그다음 글에서 VPC·Subnet·NAT Gateway 같은 네트워크 인프라를 정리했다. 이 둘을 합치면 Phase 3의 실제 작업이 남는다 — Naver Cloud에서 **NKS(Naver Kubernetes Service) 클러스터를 실제로 만드는 것**이다.

클러스터 생성 화면을 처음 열면 낯선 설정 항목이 20개 가까이 쏟아진다. 이 글은 그 항목들을 "①기본 인프라 → ②노드풀 → ③네트워크 → ④관리 안전장치 → ⑤보안 인증" 다섯 구간으로 나눠서, 지난 글에서 쓴 "아파트 단지" 비유를 이어서 정리한다. 이번엔 단지가 이미 있고, 그 안에 **새 건물(클러스터)을 착공하는** 단계라고 보면 된다.

## 1. 기본 인프라 설정

클러스터의 뼈대와 가상화 기술을 고르는 단계다.

### 하이퍼바이저 (KVM / XEN)

물리 서버 한 대를 여러 개의 가상 서버로 쪼개는 기술이다.

![Type 1 vs Type 2 Hypervisor](https://media.geeksforgeeks.org/wp-content/uploads/20251208153233886786/hypervisor.webp)

> 🏢 **비유**: 하이퍼바이저는 건물을 올릴 때 쓰는 "기초 골조 공법"이다. 특별한 이유가 없다면 요즘 가장 많이 검증된 표준 공법을 쓰면 된다.

KVM은 볼륨을 무중단으로 리사이징할 수 있고, ALB Ingress Controller가 기본 포함되어 있으며, 워커 노드를 최대 250개까지 늘릴 수 있다(XEN은 50개). 특별히 XEN을 써야 할 이유가 없다면 **KVM이 사실상 표준**이다.

### 클러스터 가용성 타입 (단일 존 / 다중 존)

![Region and Availability Zone](https://learn.microsoft.com/en-us/azure/reliability/media/availability-zones-overview/regions-availability-zones.svg)

> 🏢 **비유**: 이 단지를 "한 동네에만 몰아서 지을지, 여러 동네에 나눠 지을지"의 문제다. 단일 존은 한 동네(하나의 데이터센터)에 전부 짓는 것이고, 다중 존은 여러 동네에 나눠 짓는 것이다. 한 동네가 정전이 나도 다른 동네는 멀쩡하다.

| 구분 | 단일 존 | 다중 존 |
|------|---------|---------|
| 서버 위치 | 데이터센터 1곳 | 데이터센터 여러 곳 |
| 장애 내성 | 그 데이터센터가 죽으면 전체 중단 | 한 곳이 죽어도 서비스 유지 |
| 비용 | 낮음 | 상대적으로 높음 |
| 추천 용도 | 테스트·개발 환경 | 실제 운영(production) 환경 |

화성형 팔란티어는 공공 서비스이므로, 운영 단계에서는 **다중 존**이 기본 선택지다.

### Kubernetes 버전

> 🏢 **비유**: 이 단지에 적용할 "건축 표준 규격 매뉴얼의 버전"이다. 최신판이라고 무조건 좋은 게 아니라, 가장 많이 검증된 안정 버전(Stable)을 쓰는 게 정석이다.

베타·알파 버전은 기능은 최신이지만 운영 환경에서 검증이 덜 됐다. 별다른 이유가 없으면 **가장 최근의 Stable 버전**을 고르면 된다.

## 2. 노드풀(Node Pool) 설정

클러스터라는 "단지 부지"가 정해졌으면, 이제 그 안에 실제로 들어설 "건물 한 동"을 짓는 단계다. 이 건물 한 동이 곧 워커 노드 그룹, 즉 노드풀이다.

> 🏢 **비유**: 노드풀은 단지 안에 실제로 올라가는 "건물 한 동"이다. 건물 이름표, 골조 크기(스펙), 몇 세대(노드)를 지을지, 창고 용량(스토리지)까지 이 단계에서 정한다.

화성형 팔란티어 NKS 클러스터의 노드풀 설정값은 다음과 같이 잡았다.

| 항목 | 설정값 | 의미 |
|------|--------|------|
| 노드풀 이름 | `hspalantir-nodepool` | 이 워커 노드 그룹에 붙이는 이름표 |
| 서버 타입 | VM | Bare Metal이 아니라 가상머신 기반 워커 노드 |
| 서버 이미지 이름 | `ubuntu-24.04-nks` (유지) | 각 노드에 미리 깔리는 OS 이미지, NKS 전용 튜닝판 |
| 서버 스펙 | `[Standard] s2-g3a` (vCPU 2, Memory 8GB) (유지) | 노드 한 대의 사양 |
| 노드 수 | 2 | 워커 노드(건물 세대) 몇 채를 만들지 |
| Subnet | 자동 할당 (유지) | NKS가 알아서 적절한 서브넷에 노드를 배치 |
| Storage Size(GB) | 100 (유지) | 노드 한 대에 딸린 디스크 용량 |
| Node IAM Role | 없음 (유지) | 노드 자체에는 별도 클라우드 권한을 주지 않음 |
| Kubernetes Label | 그대로 | 노드에 붙이는 꼬리표. 특정 워크로드를 특정 노드에만 배치할 때 사용 |
| Taint | 그대로 | 자격 없는 워크로드가 이 노드에 못 들어오게 막는 출입 제한 표식 |

**왜 대부분 "유지"인가**

지금 단계(화성형 팔란티어 Phase 3 초기)에서는 트래픽 규모가 크지 않고, 검증되지 않은 값을 건드릴수록 장애 원인만 늘어난다. 노드 수 2, vCPU 2·메모리 8GB짜리 표준 스펙으로 시작해서 실제 부하를 관찰한 뒤 필요하면 노드 수나 스펙을 올리는 쪽이 안전하다. Node IAM Role을 비워둔 것도 같은 이유다 — 노드 단위로 클라우드 권한을 주기 시작하면 그 노드가 뚫렸을 때 피해 범위가 커지므로, 필요한 권한은 노드가 아니라 Pod·서비스 단위로 좁혀서 주는 게 원칙이다.

### 인증키 (PEM 파일)

노드풀을 만들 때 각 워커 노드에 SSH로 직접 접속할 수 있는 키 페어를 지정한다. 기존 키를 골라도 되고, 새로 만들면 그 순간 딱 한 번 `.pem` 파일을 내려받을 수 있다.

> 🏢 **비유**: 이 건물(노드)의 현관 마스터키다. 열쇠 복사점(Naver Cloud)은 최초 발급 시 딱 한 번만 실물 복사본을 내주고, 그 이후로는 같은 열쇠를 다시 복사해주지 않는다. 잃어버리면 그 열쇠로 여는 문 자체를 통째로 교체(키 페어 재발급 + 노드 재생성)해야 한다.

**주의사항**

- `.pem` 파일은 생성 시점에 **단 한 번만** 다운로드할 수 있다 — 재다운로드 불가, 분실하면 복구 불가
- 다운로드 직후 `chmod 400 [키파일명].pem`으로 권한을 잠가야 한다. 권한이 열려 있으면 SSH 클라이언트가 접속 자체를 거부한다
- git 저장소에는 **절대 커밋하지 않는다** — `.gitignore`에 `*.pem`을 반드시 추가
- 분실 시 해당 키로 만든 노드에는 더 이상 SSH로 들어갈 수 없다. 새 키 페어를 만들고 노드를 다시 생성해야 한다
- 담당자가 바뀌는 시점(공공사업 특성상 인사이동이 잦다)에는 키 재발급을 검토한다 — 퇴사·이동한 담당자의 PC에 키 파일이 남아있는 상태를 방치하지 않기 위해서다

## 3. 네트워크 및 서브넷 설정 ⭐

클러스터 내부의 통신 도로망을 짜는 구간이다. 여기서 잘못 설정하면 나중에 IP 대역이 충돌해서 클러스터를 다시 만들어야 할 수도 있다.

### VPC와 CNI Plugin

VPC는 지난 글에서 다룬 그 VPC다 — 클러스터가 들어갈 독립된 가상 네트워크 부지. CNI Plugin은 그 부지 안에서 컨테이너(Pod)끼리 서로 통신하게 해주는 규약이다.

![CNI Plugin Architecture](https://media.geeksforgeeks.org/wp-content/uploads/20240318123237/image-349-(1).webp)

> 🏢 **비유**: VPC가 "내가 분양받은 단지 부지"라면, CNI Plugin은 그 부지 안에 "도로를 포장하고 신호 체계를 깔아주는 시공사"다. 어느 시공사를 쓰든 목적은 같다 — 각 동(노드)과 각 세대(Pod)가 서로 오갈 수 있게 만드는 것.

### 서브넷 분리 원칙

쿠버네티스는 일반 서버용 도로(Subnet)와, 트래픽을 분산해주는 로드밸런서용 도로(LB Subnet)를 **엄격히 분리**해야 한다.

![VPC with Public/Private Subnets and Load Balancer](https://docs.aws.amazon.com/vpc/latest/userguide/images/vpc-example-private-subnets.png)

> 🏢 **비유**: 주민이 걸어 다니는 "인도"와 택배 차량이 다니는 "차도"를 반드시 분리하는 것과 같다. 사람과 차가 같은 길을 쓰면 어디선가는 반드시 부딪힌다.

**LB Public Subnet vs LB Private Subnet**

| 구분 | LB Public Subnet | LB Private Subnet |
|------|-------------------|---------------------|
| 역할 | 외부 인터넷 → 서비스 진입점 | 내부 시스템끼리(DB 등) 통신 |
| 비유 | 단지 정문 (외부 손님이 벨을 누르는 곳) | 단지 내부 전용 셔틀 정류장 (주민 전용) |
| 예시 | 화성형 팔란티어 웹 화면 진입 | Pod ↔ PostGIS 내부 통신 |

### 도커 브릿지 대역 충돌 방지

쿠버네티스 내부에서 Docker는 기본적으로 `172.17.0.0/16` 대역을 내부 통신용으로 이미 선점해서 쓴다. 그래서 우리가 새로 만드는 서브넷 주소가 이 대역과 겹치면 안 된다.

```
피해야 할 대역
172.17.0.0/16   ← Docker 내부 브릿지가 이미 사용 중

예시로 안전한 VPC 대역
10.0.0.0/16 이나 172.20.0.0/16 등, 172.17.0.0/16과 겹치지 않는 대역
```

> 🏢 **비유**: 관리사무소가 이미 단지 내부 순환버스 노선 번호로 "172-17번"을 쓰고 있다. 새로 분양하는 동/호수 체계를 이 번호랑 겹치게 정하면, 우편물(패킷)이 엉뚱한 곳으로 배달된다.

## 4. 관리 및 안전장치

### Audit Log (감사 로그)

"누가, 언제, 어떤 명령어를 내렸는지" 모든 기록을 남기는 기능이다.

> 🏢 **비유**: 단지 곳곳에 달린 CCTV와 출입 기록부다. 문제가 생기면 나중에 되돌려서 누가 어느 문을 언제 열었는지 확인할 수 있다.

해킹 방지와 사고 추적을 위해 운영 환경에서는 **설정이 사실상 필수**다. 특히 공공기관 시스템은 감사 요건 자체가 있는 경우가 많다.

### 반납 보호

누군가 실수로 '삭제' 버튼을 눌러도 클러스터가 바로 지워지지 않도록 잠금을 걸어두는 기능이다.

> 🏢 **비유**: 관리사무소 "철거 버튼" 위에 씌워둔 투명 잠금 덮개다. 커버를 열고 한 번 더 확인해야만 진짜로 누를 수 있다.

되돌릴 수 없는 실수를 막는 장치이므로 **설정을 강력히 권장**한다.

## 5. 보안 및 인증 (Access Control)

### 클러스터 인증 모드 (API vs ConfigMap)

클러스터 접근 권한(IAM)을 어디서 관리할지 고르는 항목이다.

```
API 방식 (권장)
  담당자 → Naver Cloud IAM → API로 클러스터 권한 실시간 부여/회수

ConfigMap 방식 (예전 방식)
  담당자 → 클러스터 내부 설정 파일(nks-auth)에 직접 텍스트로 권한 기입
```

> 🏢 **비유**: 단지 출입증 발급을 누가 관리하느냐의 문제다. API 방식은 관리사무소 전산 시스템이 출입증을 실시간으로 발급·회수하는 것이고, ConfigMap 방식은 경비실 수첩에 손으로 이름을 적어 관리하는 옛날 방식이다. 사람이 바뀔 때마다 수첩을 직접 고쳐 써야 하니 실수가 생기기 쉽다.

특별한 이유가 없다면 **API 방식**이 최신이고 안전하다.

### 시크릿 암호화

클러스터 내부에 저장되는 DB 비밀번호, API 키 같은 민감 정보(Secret)를 한 번 더 암호화해서 저장할지 여부다.

> 🏢 **비유**: 각 세대 금고 안에 있는 비밀번호 쪽지를, 그 금고를 통째로 더 큰 금고 안에 한 번 더 넣어 이중 잠금하는 것이다. 세대 금고(etcd)가 뚫려도 안쪽 금고(암호화 키)까지는 못 연다.

금융·공공·운영 환경이라면 **설정**이 좋다. 화성형 팔란티어는 공공사업 시스템이므로 여기 해당한다.

### ACG (Access Control Group)

클라우드 방화벽이다. 쿠버네티스 운영에 필요한 기본 포트 규칙을 Naver Cloud가 자동으로 만들어준다.

> 🏢 **비유**: 단지 정문에 관리사무소가 기본으로 세팅해주는 "표준 방문 차량 허용 목록"이다. 기본값을 그대로 쓰다가, 새로운 서비스(포트)가 필요해지면 그때 목록에 추가하면 된다.

## 6. 화성형 팔란티어 Phase 3에 적용할 값 정리

| 설정 항목 | 선택 | 이유 |
|-----------|------|------|
| 하이퍼바이저 | KVM | 표준, 무중단 볼륨 리사이징, 노드 250개까지 확장 가능 |
| 클러스터 가용성 타입 | 다중 존 | 공공 서비스라 데이터센터 한 곳 장애로 전체가 죽으면 안 됨 |
| Kubernetes 버전 | 최신 Stable | 가장 많이 검증된 버전으로 보안 패치도 최신 유지 |
| 노드풀 스펙 | `s2-g3a` (vCPU 2, Memory 8GB) × 2대 | 초기 트래픽 규모에 맞는 표준 사양으로 시작, 부하 보고 조정 |
| Node IAM Role | 없음 | 권한은 노드가 아니라 Pod·서비스 단위로 좁혀서 부여 |
| 인증키(PEM) 관리 | 다운로드 즉시 `chmod 400` + 사내 보관소 이동, git 커밋 금지 | 재발급 불가한 키라 분실 시 노드 재생성이 유일한 복구 수단 |
| VPC / Subnet | 기존 VPC 재사용, LB Subnet 별도 분리 | 지난 글에서 설계한 Public/Private 구조를 그대로 연결 |
| 서브넷 대역 | `172.17.0.0/16`과 겹치지 않는 대역 | Docker 내부 브릿지와 충돌 방지 |
| Audit Log | 설정 | 공공기관 시스템 감사 요건 + 사고 추적 |

## 정리

| 구간 | 핵심 질문 | 아파트 단지 비유 |
|------|-----------|-----------------|
| 기본 인프라 | 어떤 공법·규모로 지을까 | 골조 공법, 한 동네 vs 여러 동네 |
| 노드풀 | 건물 한 동을 어떤 사양으로 지을까 | 건물 이름표, 골조 크기, 세대 수 |
| 네트워크 | 도로를 어떻게 깔까 | 인도/차도 분리, 도로 시공사(CNI) |
| 관리 안전장치 | 실수와 사고를 어떻게 막을까 | CCTV·출입기록부, 철거 버튼 잠금 |
| 보안 인증 | 출입증을 누가 관리할까 | 전산 발급 vs 수첩 기록, 이중 금고 |

클러스터 생성 화면의 항목 하나하나는 결국 "이 단지를 얼마나 튼튼하고 안전하게 지을 것인가"라는 같은 질문의 다른 얼굴이다. Phase 1의 Vercel 서버리스에서는 이 질문 자체가 필요 없었지만, Phase 3에서 NKS로 옮기는 순간부터는 이 설정값 하나하나가 화성형 팔란티어의 안정성과 보안 수준을 결정한다.
