## 클러스터가 실제로 만들어졌다

지난 글에서 NKS 클러스터 생성 화면의 설정 항목 20여 개를 다섯 구간으로 나눠 정리했다. 그 글은 **"무엇을 고를 것인가"** 에 대한 글이었다. 이 글은 그 다음 — **"실제로 골라서 눌렀더니 어떻게 됐는가"** 에 대한 기록이다.

결론부터: `생성하기`를 누르고 10여 분 뒤 `hspalantir-cluster`가 정상 생성됐다. 중간에 에러도, 롤백도, 재시도도 없었다.

프로비저닝 자체는 이렇게 한 문장으로 끝난다. 그래서 이 글의 본론은 따로 있다. **지난 글에서 "이렇게 하겠다"고 적어둔 값과, 실제로 구축한 값이 두 군데에서 갈렸다.** 개발일지에서 가장 기록할 가치가 있는 건 계획대로 된 부분이 아니라 계획과 달라진 부분이고, 더 중요한 건 그게 왜 달라졌는지다.

## 1. 실제로 구축된 클러스터

먼저 최종적으로 만들어진 물건이 무엇인지 정리한다.

| 항목 | 값 |

| 클러스터 이름 | `hspalantir-cluster` |
| Kubernetes 버전 | 1.35.3 |
| 하이퍼바이저 | KVM |

## 2. 계획과 갈린 지점 ① — 다중 존 → 단일 존

지난 글의 "Phase 3에 적용할 값" 표에는 이렇게 적혀 있다.

> **클러스터 가용성 타입 | 다중 존 | 공공 서비스라 데이터센터 한 곳 장애로 전체가 죽으면 안 됨**

그런데 실제로는 **단일 존(KR-2)** 으로 만들었다.

### 왜 바꿨나

**지금은 테스트·개발 단계이기 때문이다.**

지난 글에서 정리한 비교표를 다시 보면, 답은 사실 그 표 안에 이미 있었다.

| 구분 | 단일 존 | 다중 존 |
|------|---------|---------|
| 장애 내성 | 그 데이터센터가 죽으면 전체 중단 | 한 곳이 죽어도 서비스 유지 |
| 비용 | 낮음 | 상대적으로 높음 |
| **추천 용도** | **테스트·개발 환경** | **실제 운영(production) 환경** |

"공공 서비스니까 다중 존"은 **운영 단계**의 요구사항이다. 지금 이 클러스터가 감당하는 건 실제 시민 트래픽이 아니라, `kubectl apply`가 되는지 / Pod가 뜨는지 / Ingress가 붙는지를 확인하는 검증 작업이다. 이 단계에서 다중 존을 켜는 건, 아직 아무도 살지 않는 모델하우스를 두 동네에 하나씩 지어놓고 관리비를 두 배로 내는 것과 같다.

> 🏢 **비유**: 다중 존은 "정전에 대비해 단지를 두 동네에 나눠 짓는 것"이다. 그런데 지금 이 단지엔 주민이 없다. 정전이 나도 불편할 사람이 없는 상태에서 이중화 비용부터 내는 건 순서가 틀렸다.

### 그럼 "공공 서비스라 다중 존"은 틀린 말이었나

아니다. **적용 시점이 빠졌던 것**이다. 지난 글의 표는 "Phase 3에 적용할 값"이라는 제목 아래 운영 기준값을 적어놨는데, Phase 3의 시작점은 운영이 아니라 검증이다. 표 한 칸에 "언제부터"가 없으면, 나중에 그 표만 보고 작업하는 사람(6개월 뒤의 나 포함)은 검증 단계에서부터 운영 비용을 태우게 된다.

그래서 이 결정은 **"운영 전환 시 다중 존 재검토"** 라는 조건과 세트로만 유효하다. 조건 없이 단일 존만 남으면 그건 계획 변경이 아니라 그냥 누락이다.

**단일 존은 클러스터를 만든 뒤에 바꿀 수 없다.** 운영 전환 시점엔 설정 변경이 아니라 클러스터 재생성이 필요하다. 그래서 지금 이 선택은 "나중에 클러스터를 한 번 다시 만든다"는 작업을 미래에 예약해 둔 것이기도 하다. 검증용 클러스터를 운영용으로 그대로 승격시킬 수 없다는 뜻이므로, 지금부터 만드는 매니페스트는 **어느 클러스터에 붙여도 동작하도록** 써야 한다. 클러스터에 종속된 값(고정 IP, 특정 노드 이름)을 매니페스트에 박아 넣으면 재생성 때 전부 손봐야 한다.

## 3. 계획과 갈린 지점 ② — 서브넷 대역

지난 글에서 도커 브릿지 대역 충돌을 경고하면서 이렇게 적었다.

```
피해야 할 대역
172.17.0.0/16   ← Docker 내부 브릿지가 이미 사용 중

예시로 안전한 VPC 대역
10.0.0.0/16 이나 172.20.0.0/16 등
```

실제로는 **192.168.0.0/16** 으로 갔다.

이건 계획 위반이 아니다. 저 예시는 "이 대역을 쓰라"가 아니라 "172.17과 겹치지 않는 대역을 쓰라"는 원칙의 예시였고, 192.168.0.0/16은 그 원칙을 만족한다. 사설 IP 대역은 세 종류가 있다.

| 대역 | 범위 | 크기 |
|------|------|------|
| 10.0.0.0/8 | 10.0.0.0 ~ 10.255.255.255 | 가장 큼 |
| 172.16.0.0/12 | 172.16.0.0 ~ 172.31.255.255 | 중간 (**172.17이 여기 포함**) |
| 192.168.0.0/16 | 192.168.0.0 ~ 192.168.255.255 | 가장 작음 |

192.168 대역을 고른 이유는 **도커 브릿지가 속한 172.16.0.0/12 대역을 통째로 피할 수 있기 때문**이다. 172.20을 쓰면 172.17과 직접 겹치진 않지만 같은 대역 안에 있어서, 나중에 다른 컨테이너 런타임이나 도구가 172.x 대역을 추가로 잡을 때 다시 확인해야 한다. 아예 다른 블록으로 가면 그 확인 자체가 필요 없다.

> 🏢 **비유**: 관리사무소가 "172-17번 순환버스"를 쓰고 있다면, 172번대 번호를 아예 안 쓰는 게 172-20번을 쓰는 것보다 확실하다. 나중에 172-25번 노선이 새로 생겨도 우리와 무관하다.

실제 서브넷 분할은 지난 글의 "인도와 차도 분리" 원칙대로 용도별로 쪼갰다.

| Subnet | 대역 | 용도 |
|--------|------|------|
| `hspalantir-public` | 192.168.1.0/24 | Public / 일반 |
| `hspalantir-private` | 192.168.2.0/24 | Private / 일반 |
| `hspalantir-nks` | 192.168.3.0/24 | Private / NKS 노드 |
| `hspalantir-lb-private` | 192.168.4.0/24 | Private / LoadBalancer |
| `hspalantir-nat-gw` | 192.168.5.0/24 | Public / NAT Gateway |

노드는 Private 서브넷에 있고, 외부 인터넷이 필요할 때는 NAT Gateway(공인 IP `223.130.131.157`)를 통해 나간다. 노드에 공인 IP를 직접 붙이지 않았으므로 외부에서 노드로 바로 들어올 수 있는 경로는 없다.

## 4. 프로비저닝이 조용히 끝난 이유

에러 없이 10여 분 만에 끝난 게 운은 아니었다. 클러스터 생성이 실패하는 흔한 원인은 대부분 **클러스터 바깥**에 있다.

| 실패 원인 | 이번에 왜 안 걸렸나 |
|-----------|-------------------|
| VPC/Subnet이 없거나 대역 충돌 | 클러스터 만들기 전에 5개 서브넷을 미리 설계·생성해둠 |
| Private 노드가 외부로 못 나감 | NAT Gateway + Route Table(0.0.0.0/0 → NAT)을 미리 연결해둠 |
| LB용 서브넷 미분리 | `hspalantir-lb-private`를 별도로 파둠 |
| 컨테이너 이미지 없음 | NCR에 `hspalantir-api:v1` 이미 push 완료 |

즉 지난 두 편(VPC·네트워크 정리 / 클러스터 설정 항목 정리)에서 한 작업이 전부 이 10분을 위한 사전 작업이었다. **네트워크를 먼저 깔고 클러스터를 나중에 만든 순서** 덕분에 프로비저닝 단계에서 할 일이 남아있지 않았던 것이다.

반대로 말하면, 클러스터 생성 화면에서 에러가 나는 경우 원인은 그 화면이 아니라 대개 그 앞 단계에 있다.

## 5. 다음 단계

클러스터는 생겼지만 아직 **말을 걸 수단이 없다.** 다음은 이 순서다.

```
1. kubectl 설치            ← winget install Kubernetes.kubectl
2. kubeconfig 다운로드      ← NCP 콘솔에서 발급
3. kubectl get nodes        ← 노드 2대가 Ready로 보이는지 확인
4. K8s 매니페스트 작성      ← secret / deployment / service / ingress
5. kubectl apply            ← 실제 배포
```

3번에서 노드 2대가 `Ready`로 뜨는 걸 확인하기 전까지는, 클러스터가 "생성됐다"는 건 콘솔 화면상의 상태일 뿐이다. 실제로 워크로드를 받을 수 있는지는 그때 확인된다.

매니페스트에는 DB 비밀번호와 API 키가 들어가므로 `secret.yaml`은 **git에 커밋하지 않는다.** 지난 글에서 `.pem` 파일에 대해 적었던 원칙이 그대로 적용된다 — 재발급 가능 여부와 무관하게, 자격증명은 저장소에 남기지 않는다.

## 정리

| 항목 | 계획 (7/14) | 실제 (7/17) | 이유 |
|------|------------|------------|------|
| 가용성 타입 | 다중 존 | **단일 존 (KR-2)** | 지금은 검증 단계. 운영 전환 시 재검토 (= 클러스터 재생성) |
| VPC 대역 | 172.17 회피 (10.x 예시) | **192.168.0.0/16** | 172.16/12 블록을 통째로 회피 |
| 하이퍼바이저 | KVM | KVM | 계획대로 |
| 노드풀 | s2-g3a × 2 | s2-g3a × 2 | 계획대로 |
| 반납 보호 / 인증 모드 | 설정 / API | 설정 / API | 계획대로 |

프로비저닝은 성공했고, 계획과 갈린 두 지점은 모두 **원칙을 어긴 게 아니라 원칙에 "언제"를 붙인 결과**였다. 다중 존은 운영 단계의 요구사항이지 검증 단계의 요구사항이 아니었고, 172.17 회피는 특정 대역을 지정한 게 아니라 충돌만 피하면 되는 원칙이었다.

설계 문서에 값을 적을 때 "무엇을"만 적고 "언제부터"를 빼놓으면, 나중에 그 문서를 읽는 사람은 검증 단계에서부터 운영 기준을 적용하게 된다. Phase 3은 아직 시작 지점이다.
