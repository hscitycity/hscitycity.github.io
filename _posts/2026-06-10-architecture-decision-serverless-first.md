## ADR-001: Phase 1 아키텍처 선택

> ADR(Architecture Decision Record)은 "왜 이 기술을 선택했는가"를 미래의 팀원(또는 미래의 자신)이 읽을 수 있게 남기는 문서다. 결정 자체보다 결정의 맥락과 트레이드오프가 중요하다.

### 결정

Phase 1(Beta v0.1)의 아키텍처를 **단일 HTML + Vercel 서버리스 함수**로 구현한다.

### 맥락

프로젝트 시작 시점(2026년 6월 초)의 상황:

- 최종 아키텍처(React + Node.js + PostgreSQL + PostGIS)는 정해져 있지만, 핵심 질문들이 아직 검증되지 않았다:
  - Naver Search API가 화성시 관련 콘텐츠를 실제로 얼마나 잘 반환하는가?
  - Claude NER로 뉴스·블로그 텍스트에서 장소명 추출이 현실적으로 가능한가?
  - AI 에이전트 응답이 도시 분석 맥락에서 실제로 유용한가?
- 공공사업 특성상 초기 시연이 빠를수록 사업 지속 가능성이 높아진다 (예산 집행 증거)
- 백엔드 인프라(PostgreSQL 서버, 도메인, SSL 등) 설정에 최소 1-2주가 필요하다

### 선택지 비교

**Option A: 단일 HTML + Vercel 서버리스** ← 선택

```
장점:
- 배포: vercel --prod 한 줄, 도메인·SSL 자동 처리
- 서버 관리 불필요 (Vercel이 스케일링·로그·배포 담당)
- CORS 문제를 서버리스 함수로 깔끔하게 해결
- 환경변수(API 키)를 브라우저에 노출하지 않음
- 프론트엔드-백엔드 코드가 한 저장소에 공존

단점:
- Vercel 서버리스 함수는 실행 시간 제한 (무료 플랜 10초, Pro 60초)
- 상태 공유 불가 (함수끼리 메모리 공유 없음)
- DB 연결이 불가능 → 데이터 영속성 없음 (조회할 때마다 외부 API 호출)
- 복잡한 비즈니스 로직 구현의 한계
```

**Option B: React + Vite + Express 서버**

```
장점:
- 최종 아키텍처와 동일 → Phase 2 전환 비용 없음
- 상태 관리, 라우팅 등 프론트엔드 기능 완전 지원
- 서버에서 DB 연결 가능

단점:
- 설정 비용: Vite 설정, ESLint, TypeScript, 번들러 설정 등 1-2일 소요
- 배포 인프라 필요: 서버, 도메인, SSL, 프로세스 매니저(PM2 등)
- Phase 1 검증 전에 "정식" 구조로 투자하는 리스크
```

**Option C: Next.js (풀스택 프레임워크)**

```
장점:
- SSR/SSG 지원, API Routes 내장
- Vercel과 최적화된 통합

단점:
- 러닝 커브: App Router vs Pages Router 선택, 서버 컴포넌트 개념 등
- 필요한 것보다 훨씬 많은 기능 → 오버엔지니어링 리스크
- 단순한 지도+API 프로토타입에 Next.js는 과하다
```

### 결정 근거

Phase 1의 목표는 **"작동하는가?"를 확인하는 것**이지, **"잘 만들어진 구조인가?"를 증명하는 것**이 아니다.

소프트웨어 엔지니어링에서 흔히 하는 실수가 검증되지 않은 가설 위에 "제대로 된" 아키텍처를 설계하는 것이다. Claude NER이 화성시 장소명 추출에서 실제로 80% 이상 정확도를 보이는지 모르는 상태에서 그것을 중심에 놓고 DB 스키마를 설계하면, 가설이 틀렸을 때 전부 바꿔야 한다.

단일 HTML + 서버리스로 2주 안에 만들어서 실제 데이터로 테스트하는 것이 낫다.

### 결과적으로 수용한 기술 부채

이 결정으로 발생하는 기술 부채는 명확하다:

1. **데이터 영속성 없음**: 검색할 때마다 Naver API를 다시 호출한다. 캐시가 없으므로 느리고 API 호출 수가 낭비된다.
2. **상태 관리 없음**: 페이지를 새로고침하면 이전 검색 결과가 사라진다.
3. **테스트 불가**: 단일 HTML에서 단위 테스트를 작성하기 어렵다.
4. **Phase 2 전환 시 전면 재작성**: Phase 1 코드를 Phase 2로 가져갈 수 없다 (아키텍처가 완전히 다름).

이 부채는 Phase 2에서 전면 재작성으로 갚는다. 의도적으로 쌓은 부채이므로 갚을 계획이 있다는 점이 중요하다.

## 서버리스 함수 구조 결정

Vercel 서버리스는 `api/` 디렉토리 아래의 JS/TS 파일이 자동으로 HTTP 엔드포인트가 된다:

```
Palantir-Demo/
├── index.html          → GET /          (정적 파일)
├── api/
│   ├── search.js       → GET /api/search
│   ├── extract.js      → POST /api/extract
│   └── agent.js        → POST /api/agent
└── vercel.json
```

`vercel.json`으로 라우팅을 명시한다:

```json
{
  "version": 2,
  "builds": [
    { "src": "api/*.js", "use": "@vercel/node" },
    { "src": "index.html", "use": "@vercel/static" }
  ],
  "routes": [
    { "src": "/api/(.*)", "dest": "/api/$1" },
    { "src": "/(.*)", "dest": "/index.html" }
  ]
}
```

세 함수의 역할 분리:

| 함수 | 역할 | 외부 의존성 |
|------|------|------------|
| `search.js` | Naver API CORS 프록시 | Naver Search API |
| `extract.js` | 텍스트 → 장소명·좌표 추출 | Claude API |
| `agent.js` | AI 대화 에이전트 (스트리밍) | Claude API |

각 함수가 단일 책임을 가지도록 설계했다. 하나의 함수가 검색+추출+에이전트를 모두 담당하면 디버깅이 어렵고 실행 시간 제한에 걸릴 가능성이 높아진다.

## CORS 문제와 서버리스의 관계

브라우저에서 Naver API를 직접 호출하면 CORS(Cross-Origin Resource Sharing) 오류가 발생한다. Naver API는 서버 간 호출을 전제로 설계되어 있어 `Access-Control-Allow-Origin: *` 헤더를 반환하지 않는다.

해결책 두 가지:

1. **CORS 프록시 서버**: 브라우저 → 내 서버(CORS 허용) → Naver API
2. **브라우저 플러그인/설정**: 개발 중에만 가능, 프로덕션 불가

`search.js` 서버리스 함수가 정확히 1번 역할을 한다. 브라우저는 내 Vercel 함수(같은 도메인)에 요청하고, 함수가 서버에서 Naver API를 호출한다.

API 키 보안도 같이 해결된다: `NAVER_CLIENT_ID`와 `NAVER_CLIENT_SECRET`은 Vercel 환경변수로 서버에만 존재하고 브라우저에는 절대 노출되지 않는다.

## 이 결정이 옳았는가

Phase 1을 마친 시점에서 돌아보면:

**맞았던 부분**: Claude NER 장소 추출의 정확도가 예상보다 낮았다. 특히 화성시 내 소지명(예: "봉담읍 왕림리")이나 신조어("동탄 호수공원 근처")의 처리가 불안정했다. 이것을 DB 스키마 설계 전에 발견했기 때문에, Phase 2에서 지오코딩 API(카카오맵 또는 네이버 지도 API)와의 하이브리드 구조를 고려할 수 있게 됐다. 만약 Phase 2 DB 설계 후에 발견했다면 스키마를 수정해야 했을 것이다.

**놓친 부분**: Vercel 무료 플랜의 실행 시간 10초 제한이 `extract.js`에서 문제가 됐다. 검색 결과 20개를 한 번에 Claude에 넘기면 평균 8-12초가 걸려서 타임아웃이 발생했다. 배치 크기를 10개로 줄이고 두 번 호출하는 방식으로 우회했지만, 처음부터 고려했어야 했다.
