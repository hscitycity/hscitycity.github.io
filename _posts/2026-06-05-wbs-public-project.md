## 공공사업 WBS는 왜 다른가

민간 스타트업에서 WBS는 주로 팀 내부 작업 분류표다. 공공사업 WBS는 다르다. 납품물(deliverable)이 계약서에 명시되어 있고, 납품물 기준으로 검수가 이루어지며, 검수를 통과해야 대금이 지급된다. 즉 **WBS = 계약 이행 증거 구조**다.

이것이 의미하는 바:

- 아무리 기술적으로 완성도가 높아도 납품 목록에 없으면 인정받지 못한다
- 반대로, 납품 목록에 있는 항목은 개발자가 비효율적이라고 생각해도 반드시 만들어야 한다
- 문서화가 개발과 같은 비중을 가진다 (설계서, 시험 결과서, 운영 매뉴얼 등)

이 현실을 이해하지 못하면 공공사업에서 번아웃된다. 기술 자체에 집중하면서도 계약 이행 증거를 동시에 관리하는 구조가 필요하다.

## 전체 WBS 구조

### 3단계 Phase 구성

```
Phase 1 (6월)    — Beta v0.1: 프로토타입, 개념 증명
Phase 2 (7~9월)  — 3-Tier: 정식 아키텍처, 핵심 기능 구현
Phase 3 (10~12월) — 확장: 외부 시스템 연계, 고도화, 이관
```

Phase 1은 납품물이 아니라 **내부 검증용**이다. "이게 실제로 작동하는가?"를 확인하기 위한 개발이다. 공공사업에서 흔히 하는 실수가 Phase 1을 건너뛰고 Phase 2부터 시작하는 것인데, 그러면 설계 단계에서 발견해야 할 문제를 구현 단계에서 발견하게 된다.

Phase 3에서 가장 중요한 것은 3.5 UX 테스트다. 팔란티어 실패 사례 분석에서 1순위 원인은 기술 문제가 아니라 "시스템은 작동하는데 공무원이 쓰지 않는 것"이었다. 담당자가 실제로 쓸 수 있는 화면을 만들지 못하면 시스템은 전시물이 된다.

## 이 WBS의 약점

솔직하게 쓰면, 이 WBS는 아직 세 가지가 불확실하다.

**1. 새올 API 연계 일정 (WBS 3.1)**

새올행정시스템은 행정안전부가 운영하는 지자체 행정 통합 시스템이다. 연계를 위해서는 행안부 보안 심의와 개인정보 영향평가(PIA)가 필요하다. 통상 3-6개월이 걸린다. 6월에 신청해도 12월 납품 시점까지 연계가 완료되지 않을 수 있다.

대응 방안: Phase 3에 배치하고, 연계 불발 시 민원 CSV 수동 업로드로 기능을 갈음한다. 납품물에는 "API 연계 설계서"를 포함시켜 작업 이행 증거를 남긴다.

**2. 국토연구원 Urban AI 모델 (WBS 3.3)**

국토연구원 Urban AI는 외부 연구팀이 개발한 민원 분석·예측 모델이다. 어떤 형태로 제공되는지(API인지 코드인지 논문인지)에 따라 시스템화 난이도가 크게 달라진다. 현재까지 모델 상세는 미확인 상태다.

**3. 민간 데이터 구매 일정 (WBS 2.3 일부)**

통신사·카드사 데이터는 도(경기도) 공동구매 또는 화성산업진흥원 개별구매로 확보한다. 예산 집행과 계약 일정이 기술 개발 일정과 맞지 않으면 데이터 없이 화면만 만들어야 한다.
