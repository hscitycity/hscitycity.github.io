## 배경: 도시 데이터는 왜 이렇게 분절되어 있는가

대한민국 지방자치단체의 데이터 현실은 생각보다 훨씬 파편화되어 있다.

화성시 하나만 해도 새올행정시스템, 국민신문고, 안전신문고, 도와드림QR, 청원24, 콜센터 민원, 당직민원 등 수십 개의 시스템이 제각각 데이터를 생산한다. 같은 지역 불법주정차 민원이 시스템마다 다른 코드, 다른 주소 표기법, 다른 처리 상태 분류로 저장된다. 환경 이슈는 환경과, 도로 민원은 건설과, 복지 수요는 복지과에 각각 고립되어 있고 어디에도 통합 뷰가 없다.

결과적으로 도시 관리자는 정확히 어떤 지역에 어떤 유형의 민원이 몇 건이나 누적되고 있는지 실시간으로 파악하지 못한다. 이것이 이 프로젝트의 출발점이다.

### 왜 "팔란티어"인가

[Palantir Technologies](https://www.palantir.com/)는 2003년 설립된 데이터 통합 분석 플랫폼 기업으로, CIA·FBI·국방부에서 시작해 지금은 영국 NHS, 우크라이나 국방부 등 복잡한 다중 출처 데이터 통합이 필요한 조직들의 의사결정 인프라로 쓰인다.

팔란티어가 증명한 것은 기술의 우수성이 아니라 하나의 명제다: **흩어진 데이터를 연결하면 의사결정의 질이 근본적으로 달라진다.**

문제는 팔란티어를 그대로 도입하면 수십~수백억 원이 필요하고, 한국 행정 체계(새올, 공공데이터포털, 민원24)와의 연계가 불가능하며, 네이버 생태계(뉴스·블로그·카페)의 데이터를 전혀 다루지 못한다는 점이다. 화성형 팔란티어는 팔란티어의 개념과 아키텍처를 참고하되, 한국 지방정부 현실에 맞게 오픈소스 스택으로 구현하는 것을 목표로 한다.

| 항목 | 내용 |
|------|------|
| 사업명 | 행정안전부-NIA '2026년 공공부문 AI서비스 지원' 공모사업 |
| 정식명 | 문제해결형 도시데이터 플랫폼 구축 |
| 주관기관 | 경기도 화성시 |
| 전문기관 | NIA (한국지능정보사회진흥원) |
| 자문기관 | 국토연구원, 화성시연구원 |
| 추진기간 | 2026년 6월 ~ 12월 (7개월) |
| 담당 | 화성시청 AI스마트전략실 |

공공사업이기 때문에 민간 스타트업과는 개발 맥락이 다르다. 기술적 완성도뿐 아니라 데이터 거버넌스, 개인정보 비식별화, 부서 간 협의체 운영, 납품 검수 기준이 동시에 요구된다. 이 시리즈는 그 과정을 있는 그대로 기록한다.

## 목표: 무엇을 만드는가

### 최종 목표 (핵심기능 7개)

사업 범위는 7개 핵심기능으로 정의되어 있다:

| 핵심기능 | 내용 | 단계 |
|---------|------|------|
| ① | 국토연구원 Urban AI 민원분석·예측 시스템 구현 | Phase 2-3 |
| ② | 온톨로지 기반 이상징후 탐지 체계 구축 | Phase 2 |
| ③ | 분석시스템 및 데이터수집 체계 구축 | Phase 1-2 |
| ④ | 정보시스템 DB 연계 (38개 → 63개) | Phase 2-3 |
| ⑤ | 타 기관 데이터플랫폼의 시 데이터 확보 | Phase 3 |
| ⑥ | 통합형 데이터표준화 및 품질관리 | Phase 2 |
| ⑦ | 문제해결형 도시데이터 GIS서비스·대시보드 구현 | Phase 1-3 |

### 현재 단계 목표 (Beta v0.1, 6월)

전체 7개 핵심기능의 기반이 되는 프로토타입을 먼저 만든다:
- 네이버 뉴스·블로그·카페에서 화성시 관련 콘텐츠 수집
- Claude AI로 텍스트에서 장소명 추출 (NER)
- 네이버 지도에 마커로 시각화
- AI 분석 에이전트로 자연어 질의응답

기술 스택: HTML + Vercel 서버리스 (서버 없이 배포 가능한 최소 구성)

## 핵심 설계 원칙: 온톨로지 기반 공간 단위

이 프로젝트의 기술적 핵심은 온톨로지다. 온톨로지란 "형식적이고 명시적인, 공유된 개념화의 명세(Studer et al., 1998)"로, 서로 다른 출처의 데이터를 공통된 개념 체계로 연결하는 기술이다.

데이터를 연결하려면 **공간 단위를 통일**해야 한다. 이 시스템은 기존 시군구·행정동 단위 대신 **국가기초구역(우편번호 5자리)**을 기본 공간 단위로 사용한다. 이유는 다음과 같다:

1. **해상도**: 행정동(평균 면적 ~2km²)보다 국가기초구역(평균 ~0.3km²)이 훨씬 세밀하다
2. **상호운용성**: 공공데이터포털, 카드사 데이터, 통신사 데이터 모두 우편번호 기반으로 제공된다
3. **안정성**: 행정동 경계는 인구 변화에 따라 자주 바뀌지만 우편번호는 비교적 안정적이다

민원 데이터의 "동탄2동", 뉴스의 "동탄신도시", 블로그의 "동탄역 근처"는 서로 다른 표현이지만 모두 같은 국가기초구역 코드로 수렴시킬 수 있다. 이것이 온톨로지의 역할이다.

## 팔란티어와의 대응 구조

| 팔란티어 플랫폼 | 화성형 대응 |
|----------------|------------|
| Gotham (다중 출처 통합 시각화) | 네이버 지도 기반 도시상황판 |
| Foundry (온톨로지·파이프라인) | 국가기초구역 공간 온톨로지 + PostGIS |
| AIP (AI 기반 의사결정 지원) | Claude API 기반 AI 분석 에이전트 |
| Apollo (자동 배포·운영) | Vercel + GitHub Actions CI/CD |

팔란티어의 핵심을 오픈소스 스택으로 복제하는 것이 목표다. 팔란티어가 수백억으로 하는 것을, 오픈소스와 클라우드 API로 어디까지 할 수 있는지가 이 프로젝트의 질문이기도 하다.

## 예상 실패 지점

공공사업 특유의 리스크와 기술적 리스크가 겹친다:

- **행정 데이터 연계 지연**: 새올행정시스템 API 연계는 법적 근거와 보안 심의가 필요하다. 협의가 지연되면 Phase 2가 늦어진다.
- **LLM 정확도**: Claude로 장소명을 추출할 때 hallucination이 발생한다. "동탄"을 경기도 화성이 아닌 다른 곳 좌표로 반환할 수 있다.
- **담당자 채택**: 팔란티어 실패 사례 1순위는 기술이 아니라 담당 공무원이 쓰지 않는 것이다. UX 설계가 기술보다 중요할 수 있다.
- **거버넌스 공백**: 데이터 정책협의체(30명 이내)가 실제로 작동하지 않으면 데이터 표준 합의가 불가능하다.

이 실패 지점들을 처음부터 알고 시작하는 것과 모르고 시작하는 것은 다르다. 실제로 어떻게 됐는지는 개발이 진행되면서 이 시리즈에 계속 기록한다.
